Hub Feed

How Pythia connects to the AncientHub node feed. Set the hub URL + the M2M HMAC secret, then Save & activate — reads fan out across the hub's advertised StoaChain nodes (see them in Observation Pool). When the feed is off, reads use the Upload Pool.

Observation Pool

The hub-fed read fleet, as advertised by the feed configured in Hub Feed. Reachability is probed from Pythia's own vantage (GET /info) — a red node can't serve reads; hover it for why. Earnings show when the hub reports them.

Pyth ledger — fleet-wide

              
What Pythia has served on StoaChain: Petitions & Pondus (keyed reads) and Transactions & Gas relayed. Counted locally; the daily Khronoton flush (ouronet-ns.PYTHIA.A_Flush) drains it on chain — watch the live per-day backlog under Pyth Flush.

              


              
Report to the hub

              
When ON, Pythia reports served usage to the AncientHub so it mints B.UNA / Stoicism to node operators. When OFF, Pythia keeps counting locally but reports nothing — that span never mints.

Unflushed backlog — what the next flush would send

              
Live per-UTC-day entries accumulating locally. This is exactly the entries[] the next A_Flush cronoton sends; on a confirmed on-chain flush these drain and counting resumes from zero. Refreshes each time you open this panel.

              

              

            


            

              
Sealed credential vault

              
Bearer credentials Pythia must use — chiefly the hub HMAC secret — are encrypted at rest (AES-256-GCM) under a master key from the deploy env (PYTHIA_MASTER_KEY), which lives off the data volume. Secret values are set in Blockchain Connectors → StoaChain → Observation Pool (they seal through this vault); the master key is an ops action, never entered here.
